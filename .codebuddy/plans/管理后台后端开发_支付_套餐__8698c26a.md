## 需求

开发管理后台缺失的后端 API，对接前端已完成的支付配置和套餐/积分管理页面。前端已使用 mock 数据开发完成，后端 ready 后替换 Service 层即可。

### 需要开发的后端模块

**1. 支付配置 API**

- 获取支付配置（微信支付/支付宝参数）
- 更新支付配置
- 数据：微信支付（enabled/appId/mchId/apiKey/apiCert）、支付宝（enabled/appId/privateKey/publicKey/gateway）、通用（currency/paymentTimeout/notifyUrl）

**2. 套餐/积分管理 API**

- 套餐 CRUD：listPlans / createPlan / updatePlan / deletePlan
- 积分配置：getCreditConfig / updateCreditConfig
- 数据：套餐（name/price/monthlyCredits/personalBudget/workspaceBudget/billingCycle/features/enabled/sort）、积分配置（pricePerCredit/minTopUpAmount/maxTopUpAmount/bonusRate/creditExpiryDays/referralRewardCredits）

## Tech Stack

- tRPC (backend procedure framework)
- Drizzle ORM (database query builder)
- Zod (input validation)
- PostgreSQL (database)

## Implementation Approach

### Architecture

沿用 workspaceAuditLog 的完整后端模式：DB Schema -> DB Model -> tRPC Router -> 注册到 lambdaRouter。

### 数据库设计

3 张表（放在一个 schema 文件中）：

**paymentConfigs**（单行配置表）

- id (text PK, nanoId(16))
- config (jsonb) — 存储完整支付配置 JSON
- updatedAt

**plans**（多行套餐表）

- id (text PK, nanoId(16))
- name, price(amountNumeric), monthlyCredits(integer)
- personalBudget (integer, optional), workspaceBudget (integer, optional)
- billingCycle (text: monthly/yearly/lifetime, default monthly)
- features (jsonb, default [])
- enabled (boolean, default true)
- sort (integer, default 0)
- createdAt, updatedAt

**creditConfigs**（单行配置表）

- id (text PK, nanoId(16))
- config (jsonb) — 存储完整积分配置 JSON
- updatedAt

### 后端分层

```
DB Schema (paymentPlans.ts)
  -> DB Model (PaymentPlansModel)
  -> tRPC Router (payment.ts + plan.ts)
  -> Lambda Router Registration (lambda/index.ts)
  -> Frontend Service (替换 mock -> 真实 tRPC 调用)
```

### 关键决策

1. 支付/积分配置使用 JSONB 单行表 — 配置项未来可扩展，无需改表结构
2. 套餐使用独立列（非 JSONB）— 需要查询/排序/校验，独立列更型安全
3. 不生成 migration 文件 — 开发阶段表会自动 sync，上线前用 drizzle-kit generate
4. 不使用 workspace scope — 这些是全局（system-wide）配置，仅 super_admin 可访问

### Routing Design

前端 tRPC 调用方式（替换 mock 后）：

- `lambdaClient.config.getPaymentConfig.query()` — 获取支付配置
- `lambdaClient.config.updatePaymentConfig.mutate({...})` — 更新支付配置
- `lambdaClient.plan.listPlans.query()` — 获取套餐列表
- `lambdaClient.plan.createPlan.mutate({...})` — 创建套餐
- `lambdaClient.plan.updatePlan.mutate({id, ...})` — 编辑套餐
- `lambdaClient.plan.deletePlan.mutate({id})` — 删除套餐
- `lambdaClient.plan.getCreditConfig.query()` — 获取积分配置
- `lambdaClient.plan.updateCreditConfig.mutate({...})` — 更新积分配置

### Performance

- 单行配置表每次全量读写（数据量极小，< 1KB）
- 套餐表预计条目数极少（10-20 条），一次全量查询无压力
- 所有操作走存储过程直接执行，无需缓存

### Blast Radius Control

- 不修改任何现有文件的功能逻辑
- 只新增 4 个文件，修改 2 个注册文件
- Schema 仅用于管理后台，不影响主应用用户
