## 2026-07-08 修复 Admin 系统设置保存失败问题

### 根因

前端 `adminSettingsService.updateSystemConfig()` 调用 `lambdaClient.config.updateSystemConfig.mutate(input)`，但后端 `configRouter` 中没有注册 `updateSystemConfig` procedure（`businessConfigEndpoints = {}` 空对象），tRPC 返回 NOT_FOUND 错误。

### 修改文件

1. **`packages/business-server/src/lambda-routers/config.ts`** — 添加 `updateSystemConfig` mutation：使用 `adminGuardProcedure` 权限控制，将设置保存到 `dict_configs` 表（group='system_settings'），支持 upsert 逻辑
2. **`apps/server/src/routers/lambda/config/index.ts`** — `getGlobalConfig` 添加 `serverDatabase` 中间件，读取 `dict_configs` 中 `system_settings` 组的配置并合并到返回结果中

### 数据流

- **保存**: UI → `handleSave` → `updateSystemConfig` mutation → `dict_configs` 表（key/value 格式）
- **加载**: `getGlobalConfig` query → 环境变量配置 + `dict_configs` 中 `system_settings` 组配置 → 返回给前端

## 2026-07-08 全面修复 Admin 保存问题（Decoration + Workspace）

### 1. Decoration 页面保存修复

- **根因**: `updateSystemConfig` 的 Zod schema 只定义了 4 个字段，但 Decoration 页面传入了 `systemLogo/favicon/themeColor/primaryColor/neutralColor` 等额外字段。默认 `z.object()` 会 strip 未知字段，导致这些值未保存。
- **修改**: `packages/business-server/src/lambda-routers/config.ts` — Zod schema 添加 5 个新字段；`src/services/admin/settings.ts` — `UpdateSystemConfigParams` 类型添加对应字段

### 2. Workspace 页面创建/操作修复

- **根因**: `packages/business-server/src/lambda-routers/workspace.ts` 是空壳 stub（仅 `ensureMarketOrganization`），`workspaceMemberRouter` 是空路由。所有 workspace CRUD 操作返回 NOT_IMPLEMENTED。
- **修改**:
  - `packages/business-server/src/lambda-routers/workspace.ts` — 实现完整 CRUD（list/getById/create/update/delete），使用 `adminGuardProcedure` 权限控制，直接操作 workspaces 表
  - `packages/business-server/src/lambda-routers/workspaceMember.ts` — 实现 `list` query 用于详情页显示成员列表
