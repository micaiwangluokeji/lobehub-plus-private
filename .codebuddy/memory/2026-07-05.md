## 全面调研报告（2026-07-05）

### 核心发现：三套独立用户分层体系

1. **RBAC 角色**（真正运行）：super_admin/vip_user/free_user，权限中间件活跃
2. **订阅计划**（框架存在）：subscriptions/plans 表有 CRUD，但 `getSubscriptionPlan()` 是 stub（硬编码返回 Free）
3. **会员等级**（仅管理端 CRUD）：membership_levels 表 + admin UI，无自动升级/权益执行

**三者之间无任何业务逻辑连接。** `vip_user` 角色从未被 `assignSystemRoleToUser` 分配，`initNewUserForBusiness` 是空函数。

### Free User 权限极度受限
- 当前 free_user 只有 3 个权限码（agent:read/update/delete:owner）
- 不能创建 session/message（不能聊天），不能调用 AI 模型，不能上传文件
- 建议放宽到包含基础聊天 + 系统默认模型调用

### 数据隔离
- 所有核心数据表（agents/files/sessions/messages/knowledge_bases/tasks/memories）都通过 userId+workspaceId 双重隔离
- 所有设置页面通过 `user_settings` 表（PK=users.id）用户隔离
- Discover 页面展示的 official agent 全局可见，但安装后副本归用户所有

- `docs/research/membership-rbac-data-isolation-survey.md` — 383 行完整调研报告
- 覆盖：会员等级 vs 套餐订阅冲突分析、Free User 权限全景、11 个页面数据隔离验证、Feature Flags、商业化改造建议

## LifeOS 调研 + 商业化 PRD v2.0（2026-07-05 下午）

### LifeOS 商业模式关键发现
- LifeOS 用 **Power（算力）** 作为统一虚拟货币，充值获得/AI调用消耗
- Plan ⊃ Level 嵌套关系：Plan 是产品分组（标准版/企业版），Level 是具体等级+价格
- 新用户注册后零权限零算力（纯白名单），需要管理员手动分配
- 权限守卫：AuthGuard → PermissionsGuard → SuperAdminGuard → MemberOnlyGuard
- LifeOS 没有 workspace 概念，LobeHub 的 personal/workspace 和 Plan 是不同的维度

### PRD v2.0 产出
- `docs/research/lobehub-commercial-v2-prd.md` — 634 行完整 PRD + 开发计划
- 核心设计：积分(Credits)=统一货币，Free→Pro 升级路径，邀请奖励体系
- 权限前后端交互逻辑：完整时序图（前端 usePermission 乐观允许 + 后端 RBAC 中间件守门 + 积分余额检查）
- 6 个开发阶段：RBAC重构 → 积分消费 → 注册赠送+邀请 → 订阅→Pro桥接 → 用户端原生页面 → 集成测试
- 新角色 pro_user：替代从未使用的 vip_user，权限 = free_user 基础 + agent:create/fork/publish + provider:create
- Free user 权限大幅放宽：补充 session/message/file/document/knowledge_base/ai_model 权限
- 附 10 条验收标准、完整文件变更清单
