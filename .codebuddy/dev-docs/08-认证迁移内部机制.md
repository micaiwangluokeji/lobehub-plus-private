# 认证迁移内部机制

> **来源**：https://lobehub.com/zh/docs/self-hosting/migration/v2/auth/migration-internals
> **所属**：自托管 > 迁移 > V2 > 认证

## 核心数据库 Schema

### `users` 表

与认证框架无关的通用用户表。
关键字段：`id`(PK)、`email`(UNIQUE)、`avatar`、`firstName`、`lastName`

### `accounts` 表

存储不同认证提供商（SSO、邮箱密码等）的连接信息。
每个独立登录方式被视为一个独立的"账号"，通过 `user_id` 外键关联到 `users` 表。
关系：一个用户可以有多个账号（1:N 关系）。

## 迁移原理

迁移过程**不会保留**用户的登录会话，迁移后所有用户必须重新登录。

### 简单迁移

仅迁移 `users` 表，忽略 `accounts` 表。

- **工作原理**: 用户重新登录时，Better Auth 通过邮箱查找 `users` 表。如果找到匹配的用户，则在 `accounts` 表中创建一条新账号记录。
- **局限性**: 系统无法识别用户之前关联的其他账号。使用副邮箱登录会创建一个新用户。

### 完整迁移

迁移 `users` 表和 `accounts` 表。

- **NextAuth**: 将 `nextauth_accounts` 表数据迁移到 Better Auth 的 `accounts` 表
- **Clerk**: 从 Clerk API 获取 `externalAccounts` 数据迁移

## 问题排查

### 登录创建了新用户而不是关联

1. 在 `users` 表中通过主邮箱找到原始用户，记录 `id`
2. 在 `accounts` 表中查询该用户关联的账号
3. 删除错误创建的 `accounts` 和 `users` 记录
4. 使用主邮箱登录，重新关联副账号

### SSO 登录不工作

1. 确认 SSO 提供商已在 `AUTH_SSO_PROVIDERS` 中配置
2. 验证提供商凭证是否正确
3. 检查 `accounts` 表中的账号记录是否创建正确
